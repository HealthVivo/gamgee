11  /*
12  * VariantHeaderMerger class
13  * @brief Class for merging VCF headers and maintain field and sample mappings
14  * Class which:
15  * (a) Merges multiple input VCF headers to obtain a merged VCF header
16  * (b) Creates and updates mappings between input VCFs and merged VCF header fields and samples.
17  *
18  * Provides functions to add new VCF headers for merging
19  * Example usage:
20  * shared_ptr<bcf_hdr_t> vcf_hdr;
21  * vector<shared_ptr<bcf_hdr_t>> hdr_vec1;
22  * vector<shared_ptr<bcf_hdr_t>> hdr_vec2;
23  *
24  * Start with empty object, add headers later
25  * VariantHeaderMerger X;
26  * X.add_header(vcf_hdr);
27  * X.add_headers(hdr_vec1);
28  * const shared_ptr<bcf_hdr_t>& merged_hdr = X.get_merged_header();
29  *
30  * Start with 1, add more later
31  * VariantHeaderMerger Y { vcf_hdr };
32  * Y.add_headers(hdr_vec1);
33  *
34  * Start with many, add more later
35  * VariantHeaderMerger Z { hdr_vec1 };
36  * Z.add_headers(hdr_vec2);
37  *
38  * The class contains two LUTs - m_header_fields_LUT and m_samples_LUT of type MergedVCFLUTBase<> for storing mapping for
39  * header fields (FMT, FLT, INFO) and samples respectively. The class is templated to select the 'best' memory layout.
40  *
41  * Each of the two MergedVCFLUTBase<> objects (m_header_fields_LUT, m_samples_LUT) contains two matrices (vector<vector<int>>):
42  * m_inputs_2_merged_lut and m_merged_2_inputs_lut. The first stores the mapping from input VCF fields to the merged VCF fields while
43  * the second stores the mapping in the opposite direction.
44  * You can layout each matrix in one of the 2 following ways:
45  * (a) matrix[i][j] corresponds to input VCF i and field j
46  * (b) matrix[i][j] corresponds to field i and input VCF j
47  * Option (a) is optimal where you are looking at all the fields of a VCF in quick succession,
48  * while (b) is optimal when you are looking at all VCFs for a particular field.
49  * The 2 boolean template parameters for MergedVCFLUTBase<inputs_2_merged_LUT_is_input_ordered, merged_2_inputs_LUT_is_input_ordered>
50  * control the layout of the two matrices. If the parameter inputs_2_merged_LUT_is_input_ordered is true, then layout (a) is selected for
51  * m_inputs_2_merged_lut, else layout (b)
52  *
53  * VariantHeaderMerger has 4 boolean parameters - two for the m_header_fields_LUT (fields_*_LUT_ordering) and two for the
54  * m_samples_LUT (samples_*_LUT_ordering)
55  * fields_forward_LUT_ordering: controls the layout of the input VCF fields to merged VCF fields matrix in m_header_fields_LUT
56  * fields_reverse_LUT_ordering: controls the layout of the merged VCF fields to input VCF fields matrix in m_header_fields_LUT
57  * samples_forward_LUT_ordering: controls the layout of the input VCF fields to merged VCF samples matrix in m_samples_LUT
58  * samples_reverse_LUT_ordering: controls the layout of the merged VCF samples to input VCF samples matrix in m_samples_LUT
59  *
60  * We create type aliases InputOrderedVariantHeaderMerger and FieldOrderedVariantHeaderMerger for specialized template
61  * instantiations of VariantHeaderMerger.
62  * (a) InputOrderedVariantHeaderMerger - fast traversal of all field lookups for a single input VCF
63  * (b) FieldOrderedVariantHeaderMerger - fast traversal of all input VCFs for a single field lookup
64  *
65  * Example usage:
66  * vector<VariantHeader> hdr_vec;
67  * VariantHeaderMerger Z { hdr_vec };
68  * auto input_vcf_idx = 0u;
69  * auto input_PL_idx = hdr_vec[input_vcf_idx].field_index("PL", BCF_HL_FMT);
70  * auto merged_PL_idx = Z.get_merged_header_idx_for_input(input_vcf_idx, input_PL_idx);
71  * auto refind_input_PL_idx = Z.get_input_header_idx_for_merged(input_vcf_idx, merged_PL_idx);
72  * assert(refind_input_PL_idx == input_PL_idx);
73  */
